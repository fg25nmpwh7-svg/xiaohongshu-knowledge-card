# 小红书知识卡片生成器

把用户当天学到的零散知识点，变成「有强观点、能传播、能收藏」的小红书图文卡片，并在 Figma 里排好版、导出 PNG。

这个 skill 解决两个独立又衔接的问题：**内容**（写什么）和**呈现**（怎么排）。两者可以分开做——用户可能只要文案，也可能要成品图。先问清楚要到哪一步。

## 工作流

1. **确认产出边界**：只要文案（Markdown）？还是要 Figma 成品 + PNG？默认先给文案，确认风格后再建图。建图是外部服务调用，动手前一句话告知。
2. **选内容形态**（见下方"两种卡片形态"）：信息流式多页卡，还是发布会式单页卡。
3. **写内容**：严格按 `references/content-style.md` 的风格——强观点、反直觉、判断句+支撑句、金句。先在对话里出文案让用户确认，**不要**没确认就直接建图。
4. **Figma 建图**：按 `references/figma-build.md` 的模板和 API 注意事项执行，避开已知的坑。
5. **导出 PNG**：`download_assets` 拿临时 URL → 立刻 `curl` 下载到本地 `exports/`（URL 会过期）。
6. **（可选）发布文案**：用户要发小红书时，给标题（≤20字带钩子）+ 正文 + 话题标签。

## 两种卡片形态

根据用户要的风格选，或直接问。

### A. 信息流多页卡（默认，强传播）
每张主题 = 1 个封面 + 5 页正文（问题 → 常见认知 → 为什么错 → 正确认知 → 金句）。每张一个主色，封面和金句页用纯色反白大字。适合知识科普、强观点输出。

### B. 发布会式单页卡（极简科技感）
每个主题 = 1 张卡：eyebrow 小标签 + 大标题 + 3 个核心观点（**判断句 + 灰色支撑句**两层）+ 底部金句。纯黑底、单一蓝色（#2997FF）点缀、大留白。苹果 keynote 调性。

> 关键经验：单页卡如果每个观点只有一句话，会显得"太干"。一定用**判断句（白色大字）+ 支撑句（灰色小字，带具体例子/机制/数字）**两层，信息密度才够，又不破坏留白。

## 尺寸与封面安全区

- 卡片统一 **1080 × 1440（3:4）**。小红书信息流封面默认按 3:4 裁切，3:4 的图正好不被裁。
- 多页卡的第 1 张是封面，决定点击率；其余按顺序左滑递进，导出和上传都别打乱顺序。

## 内容硬要求（来自项目协作规范）

- **禁止空话套话**。每个观点配至少一个具体例子：真实字段名、示例数值、实际输入/输出。
- 反直觉、强判断优先于"知识点罗列"。少定义、多判断；少解释、多洞察。
- 金句 ≤20 字，能单独截图当封面传播。

详见 `references/content-style.md`。

## Figma 建图：必读注意事项

`use_figma` 跑的是 Figma Plugin API 的 JS，有几个**每次都会踩的坑**，建图前先读 `references/figma-build.md` 拿可直接用的模板。最关键的几条：

- 写文字前**必须先 `loadFontAsync`** 所有用到的字重，否则报错。
- `figma.currentPage = page` **不支持**，要用 `await figma.setCurrentPageAsync(page)`。
- `figma.loadAllPagesAsync()` **不支持**；读另一个 page 的 children 前要先 `setCurrentPageAsync` 切过去，否则 `page.children` 是空数组。
- 中文用 `Inter` 字体即可，靠 Figma 内置 CJK 回退渲染，导出 PNG 中文正常显示。**提醒用户**：要更精致可换思源黑体/Noto Sans SC 后重导。
- `create_new_file` 需要 `planKey`，先 `whoami` 拿。View 席位也能编辑自己 drafts 里的文件。

## 导出 PNG

`download_assets`（传 `nodeId`，`defaultFormat: png`）返回的是**临时 URL**，要立刻下载：

```bash
curl -s -o exports/Card-1.png "<临时URL>"
file exports/Card-1.png   # 验证：应为 PNG 1080 x 1440
```

拿 nodeId 的方法：建完图后在同一段 `use_figma` 里 `return frames.map(f=>({name:f.name,id:f.id}))`。

## 发布说明

不替用户自动发布——小红书无开放发图 API，全自动靠模拟点击会有限流/封号风险，且登录扫码、验证码不能代过。要发布时给**半自动**方案：用浏览器工具帮填图+文案，登录/验证码/最后点"发布"留给用户。
